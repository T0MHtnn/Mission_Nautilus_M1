# 1.4. Rapport de Performance - TP6 - 2.1

## 1. Description des fonctionnalités intégrées
*   **Filtrage des Rivaux** : Le store exclut désormais les joueurs ayant le rôle "rival" lors de la réception des données via un `.filter()`. Cela sécurise l'interface pour les Explorateurs.
*   **Mouvement continu** : Implémentation d'une animation fluide via `requestAnimationFrame`. Les objets ne "sautent" plus lors du rafraîchissement des données API toutes les 5 secondes.
*   **Incertitude géographique** : Chaque objet non découvert se voit attribuer une cible aléatoire. Une fois la cible atteinte, une nouvelle est générée au polling suivant, créant un mouvement d'errance perpétuel.

## 2. Mesures User Timing (Post-développement)

| Métrique | Valeur (v0) | Valeur actuelle (2.1) | Écart |
| :--- | :--- | :--- | :--- |
| **Synchronisation Ressources** | 0.1 ms | **0.5 ms** | + 0.4 ms |
| **Traitement Logique Jeu** | 4.3 ms | **4.0 ms** | - 0.3 ms |

> **Analyse :** 
> L'augmentation du temps de synchronisation des ressources est directement liée à l'ajout de la boucle de recherche (`find`) et à la gestion de la `Map` des cibles. Le temps global de traitement reste cependant extrêmement faible, n'induisant aucun ralentissement de la logique métier.

## 3. Rapport Lighthouse (Mobile - Performance uniquement)

*   **Score Global de Performance** : 55 / 100
*   **Total Blocking Time (TBT)** : 100 ms

> **Observations :**
> Malgré l'ajout d'une boucle d'animation à 60 FPS, le **Total Blocking Time** reste très faible (100 ms), ce qui indique que les calculs d'interpolation (LERP) sont suffisamment légers pour ne pas saturer le thread principal. L'amélioration du score global (55) par rapport aux étapes précédentes suggère une meilleure stabilité du thread principal lors des phases de repos entre les requêtes API, validant ainsi l'utilisation de `requestAnimationFrame` pour la fluidité visuelle.
